# Инфраструктура

Это не база знаний по контекстной рекламе (для этого — `База_знаний/`) и не лог
решений по конкретному клиенту (для этого — `Клиенты/<клиент>/Решения.md`).
Здесь — история решений о том, как устроена сама система: вики, скрипты,
автоматизация. Формат — как везде: `## [ГГГГ-ММ-ДД] тема | решение | почему`.

Если тут наберётся много разноплановых страниц (не просто хронология, а
несколько независимых тем) — превратить в папку `Система/` со своим index.md.
Пока хватает одного файла.

## [2026-07-16] Скрипты сбора статистики (Директ / Метрика)

Расположение: `Скрипты/direct_stats.py`, `Скрипты/metrika_stats.py`, общий модуль
`Скрипты/_config.py`. Подробности запуска — `Скрипты/README.md`.

**Решение:** один скрипт на источник данных (не по скрипту на клиента) —
различие между клиентами только в логине/токене/номерах целей, структура
запроса одинаковая.

**Почему:** логин/токен/счётчик Метрики не дублируются в этой вики — читаются
напрямую из общей Google-таблицы (`1ymDNHkj32mYIb_ymf7t8O6H6WY1s5F1FN43-56fwiQM`,
вкладки "Директ"/"Metrika"), той же самой, что питает продовый BigQuery-пайплайн
в `E:/PythonProjects/RedBird/google-cloud-jobs`. Единый источник правды — если
токен обновится там, здесь ничего менять не нужно. Доступ — через service
account `Скрипты/secrets/rb_cloud_service.json` (скопирован оттуда же, не в git,
read-only к таблице).

Логика запроса адаптирована из `google-cloud-jobs/redbird/func/direct_stats_job_src`
и `metrika-stats-job_src`, но без BigQuery/Google Sheets записи — работает на
одного клиента за раз, кладёт TSV/CSV снапшот прямо в
`Клиенты/<клиент>/Статистика/`.

**Проверено:** 2026-07-16 — успешный запуск на клиенте [[ЕГЭ Merlin]] (реальные
данные по кампаниям и визитам получены).

**Открытый вопрос:** пока запуск ручной. Если станет неудобно — обсудить
автоматизацию (cron/Task Scheduler) с фиксированной периодичностью снапшотов.

## [2026-07-16] Разделение База_знаний/Инструменты vs Инфраструктура

**Решение:** `База_знаний/Инструменты/` — строго только отраслевое знание
(фичи/инструменты Директа и Ads), извлечённое из статей и мануалов, которые
приносит пользователь, со статусом протестировано/не протестировано на наших
кампаниях. Документация про наши собственные скрипты и устройство вики туда не
попадает — для этого этот файл (`Инфраструктура.md`) и `Скрипты/README.md`.

**Почему:** изначально страница про скрипты сбора статистики была ошибочно
положена в `База_знаний/Инструменты/` — смешала внешнее отраслевое знание с
внутренней документацией инструментов. Это две разные вещи с разной судьбой:
знания из статей растут от источников и не меняются, когда меняется наш код;
факты про нашу инфраструктуру меняются, когда меняется наш код, и не зависят от
новых статей.

## [2026-07-16] Схема целей дашборда: Директ (расход/клики) + Метрика (цели) + колл-трекинг (звонки) напрямую

**Решение:** так же, как в проде — расход и клики берём из Директа, конверсии
(заявки через сайт) — из целей Метрики, а звонки, если у клиента подключён
колл-трекинг — из API колл-трекинг-сервиса напрямую, а не из целей Метрики.

**Почему:** источник трафика "Яндекс.Директ: не определено" в Метрике иногда не
долетает до правильной атрибуции — часть звонков просто не попадёт в отчёт,
если брать их только из целей Метрики. Это видно в таблице (вкладка "Metrika")
по колонкам `goal_id` (обычная цель Метрики) / `ct_goal_id` (цель
колл-трекинга) / `ct_goal_source` — если `ct_goal_source` заполнено (например,
`api`), звонки нужно брать из API этого сервиса, а не из Метрики.

**Открытая задача:** у нас пока нет скрипта, который тянет звонки из API
колл-трекинга (для ЕГЭ Merlin это Callibri, см. [[Клиенты/ЕГЭ Merlin/Цели/goals|Цели ЕГЭ Merlin]]).
Из-за этого текущий анализ по заявкам неполный — метрика-конверсии видим, звонки
нет. Нужно писать `callibri_stats.py` по аналогии с `direct_stats.py` /
`metrika_stats.py`, когда будем разбираться со звонками для реального KPI.

## [2026-07-16] Имена файлов снапшотов с фильтром по целям

**Решение:** если `direct_stats.py`/`metrika_stats.py` вызваны с `--goals`, имя
выходного файла получает суффикс `_goals-<id1>-<id2>`.

**Почему:** без суффикса повторный вызов с другим набором `--goals`, но тем же
диапазоном дат молча перезаписывал предыдущий снапшот (уже случилось один раз —
потеряли полный снапшот Метрики, пришлось перезапрашивать).
