# Сверка конверсий: Директ + Метрика + Колл-трекинг

Применяется ко всем проектам, не только к [[../../Клиенты/ЕГЭ Merlin/index|ЕГЭ Merlin]]
(там впервые всплыл повод это формализовать — см. `Решения.md` от 2026-07-16).

## Базовый принцип

Проще и правильнее всего тянуть **расход, клики И конверсии одним запросом к
Директ API** (параметр `Goals`/`AttributionModels`, номера целей — из вкладки
"Metrika" общей таблицы). Это даёт максимальную гранулярность: разбивку по
ключевым словам, полу, возрасту, региону, устройству — то, чего Метрика сама по
себе в таком разрезе не даёт.

## Почему в Директ попадают не все конверсии

1. **Метрика иногда фиксирует только канал, без остального.** Конверсия
   привязана к каналу "Яндекс.Директ: не определено" (формулировка уточняется),
   то есть источник распознан как реклама Директа, но кампания/ключевое слово —
   нет. UTM-метка (в частности UTM Campaign) при этом в Метрике может всё же
   сохраниться — тогда кампанию можно восстановить вручную по ней, даже если
   штатная привязка в Директе не сработала.
2. **Колл-трекинг передаёт в Метрику не все звонки.** Например, в самом
   колл-трекинге (Callibri и т.п.) зафиксировано 3 звонка, а в Метрику дошло 2 —
   по разным техническим причинам на стороне интеграции.

## Как сверять (перед стратегическими решениями по кампании)

1. Взять конверсии из Директа (через `Goals`) — это база.
2. Отдельно взять конверсии из Метрики + звонки напрямую из API колл-трекинга,
   объединить через UTM-метки (кампания как минимум; ключевое слово/регион/
   устройство — если метка это передаёт).
3. Сравнить: конверсии по Директу vs объединённые Метрика+колл-трекинг —
   **хотя бы на уровне кампании** (комбо).
4. Если расхождение небольшое — работать на данных Директа, ничего не
   пересчитывать.
5. Если расхождение большое — разбираться, как учитывать это в решении по
   кампании/оптимизации, ориентируясь на более полную (объединённую) цифру, но
   **только на том уровне детализации, где сверка вообще возможна** (см. ниже).

## Жёсткое ограничение точности

- Через UTM-метки сверка возможна максимум до уровня кампании (иногда —
  ключевое слово, регион, устройство, если метка это передаёт).
- **Пол и возраст через UTM не передаются вообще** — сверить на этом уровне
  детализации невозможно в принципе, не только сложно.
- Даже склейка на уровне кампании не гарантированно точна — есть риск
  привязать конверсию не к тому визиту (похоже, из-за неоднозначностей с
  client/user ID). Объединённая цифра точнее цифры Директа только в среднем, не
  построчно.

## На каком уровне детализации принимать решение по итогам сверки

Если сверка показывает большое расхождение по кампании (Директ занижает), это
не значит, что можно так же уверенно резать решение по любому более узкому
срезу внутри неё. Критерий: смотреть, долетает ли до среза UTM-метка.

- **Срез восстановим по UTM** (гео, устройство, группа объявлений, объявление,
  ключевое слово — если параметр присутствует в utm_content/utm_term) —
  сверять и принимать решение на этом уровне детализации, так же как на уровне
  кампании.
- **Срез принципиально не передаётся через UTM** (пол, возраст — см. "Жёсткое
  ограничение точности" выше) — сверка на этом уровне невозможна в принципе, не
  только сложно. В этом случае решение принимать **только на уровне кампании
  целиком**, не разбивая по такому срезу вслепую на одних заниженных данных
  Директа: нет способа понять, теряются ли скрытые конверсии равномерно по
  срезу или сконцентрированы в одной его части.

Пример: Директ показывает по кампании 2 конверсии дорого, сверка (Метрика+
Callibri) показывает 6 конверсий в рамках KPI — по кампании в целом вывод
"работает" на объединённой цифре обоснован. Но урезать/расширять внутри неё
конкретно по полу/возрасту на основании одного лишь Директа — нельзя, там
эта потеря конверсий может быть распределена как угодно.

## Правило на случай, если сверка не получается

Если объединить данные через UTM надёжно не получается (нет доступа к API
колл-трекинга, в Метрике потеряна и метка тоже, слишком много сомнительных
совпадений при склейке) — **не пытаться натянуть шаткое объединение ради
видимости точности**. Такая склейка может добавить собственную ошибку вместо
того, чтобы её убрать. В этом случае — осознанно работать на данных Директа как
есть, принимая, что они, возможно, немного занижены. Это компромисс, а не
пробел, если он явно признан.
