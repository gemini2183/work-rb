Кампания "мк / егэ / набор 26" (710926114) — расхождение большое: Директ показывает 1 конверсию за весь период, по факту (Метрика+Callibri) — 4. Дополнительно у этой кампании подозрительная картина по расходу: 693 клика за 26 дней, из них расход показан только за один день (23 июня, 2257₽), все остальные дни — 0,00₽ при ненулевых кликах. Предварительно объяснили это оплатой за конверсию (PAY_FOR_CONVERSION) — при такой стратегии расход по дням действительно может быть неравномерным. Но подтвердить через API не удалось: кампания 710926114 не видна через Campaigns.get под нашим логином/токеном вообще (пусто даже с явным SelectionCriteria.Types=[UNIFIED_CAMPAIGN] и прямым запросом по Id). Уточнили: Мастер кампаний — не отдельный тип API, а надстройка, создающая обычные TEXT_CAMPAIGN/UNIFIED_CAMPAIGN на бэкенде, так что дело не в типе. Проверка показала: Campaigns.get под этим логином вообще не возвращает НИ ОДНОЙ кампании с Id больше ~708 млн (максимальный видимый Id — 708176298, без обрезки пагинацией — LimitedBy пуст) — то есть недоступны все кампании новее какой-то даты/порога, включая даже letnie-кампании (710870578, 712259476), которые точно активны и видны в Метрике. Причина прояснена: 710926114 и letnie-кампании — не Мастер кампаний как надстройка над TEXT_CAMPAIGN/UNIFIED_CAMPAIGN, а Товарная галерея/Товарный мастер — упрощённый вид кампаний, чья конфигурация в Директе read-only и не отдаётся через Campaigns.get в принципе (проверили и SmartCampaignFieldNames с Types=[SMART_CAMPAIGN] — тоже пусто). Это архитектурное ограничение API, не баг запроса и не проблема прав логина. Настройки/стратегию таких кампаний можно посмотреть только в веб-интерфейсе; статистика (Reports API) при этом полноценно доступна и уже используется. Вывод: расход "0₽ при кликах" по 710926114, скорее всего, объясняется оплатой за конверсию (PAY_FOR_CONVERSION) — типичное поведение для такой стратегии, но финальное подтверждение — за проверкой в интерфейсе.
