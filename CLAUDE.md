# Схема хранилища: управление проектами по контекстной рекламе

Это Obsidian-вики по паттерну "LLM Wiki". Ты — единственный, кто пишет и поддерживает
структуру. Пользователь курирует источники, задаёт вопросы и принимает решения; вся
рутина (кросс-ссылки, обновление статусов, ведение логов) — твоя ответственность.

## Структура

- `Клиенты/<Имя>/` — рабочая папка по каждому проекту (клиенту). Шаблон — в
  `Клиенты/_шаблон_клиента/`, копируй его при заведении нового клиента.
  - `_project.md` — бриф: ниша, ЛПР, история отношений, договорённости.
  - `Цели/goals.md` — цели аналитики (Метрика/GA): ID, название, макро/микро, что считается конверсией.
  - `Семантика/` — мастер-список ключей: статус (в проработке / использован / в архиве / отклонён), в какой кампании.
  - `Аудитории/` — сегменты ретаргетинга: где заведены, на каком этапе воронки, статус проработки по типам кампаний.
  - `Кампании/Search/`, `Кампании/РСЯ_КМС/` — по файлу на кампанию. Внутри — append-only лог действий и эффектов.
  - `Статистика/` — снапшоты, которые кладут скрипты из `Скрипты/`. Ты их читаешь и анализируешь тренды, сам туда не пишешь руками.
  - `Решения.md` — append-only стратегический лог клиента (формат ниже).
  - `index.md` — точка входа по клиенту: ссылки на всё выше + текущий статус одной строкой.
- `База_знаний/` — общие, кросс-клиентские знания (не привязаны к одному проекту).
  - `Источники/` — сырые материалы (статьи, доки по инструментам). Неизменяемые, только читаешь.
  - `Инструменты/` — вики-страницы по фичам/сервисам Директа и Ads. Frontmatter `status: протестировано|не протестировано|отклонено`.
  - `Паттерны/` — обобщённые выводы ("что работает / не работает"), со ссылками на конкретные записи в `Решения.md` клиентов как на доказательство.
  - `index.md`, `log.md` — как в паттерне Karpathy.
- `Бэклог_идей.md` — append-only список идей и гипотез, всплывших в разговоре, которые не относятся к текущей задаче, но жалко терять. См. правило ниже.
- `Скрипты/` — Python-инструменты, тянущие статистику из API Директа/Ads в `Статистика/` клиента. Скрипты только собирают данные, не делают выводов.

## Формат записей в логах (Решения.md, лог кампании)

Единый парсибл-префикс:

```
## [ГГГГ-ММ-ДД] Кампания/тема | действие | гипотеза | результат
```

Это позволяет `grep "^## \["` по всем клиентам разом для еженедельного обзора.

## Протокол в начале сессии по клиенту

1. Прочитать `Клиенты/<Имя>/index.md` и `_project.md`.
2. Прочитать последние ~10 записей `Решения.md`.
3. Если вопрос касается конкретной кампании/семантики/аудитории — открыть соответствующий файл.
4. Проверить `База_знаний/Паттерны/` на релевантные обобщения, прежде чем предлагать стратегию с нуля.

## Протокол на вопрос по практике/инструментам

1. Сначала искать в `База_знаний/Инструменты/` и `Паттерны/`.
2. Если там нет ответа — явно сказать пользователю "в базе знаний этого нет, ищу внешне",
   пойти в WebSearch/память модели, ответить.
3. После ответа — обязательно оформить результат новой или обновлённой страницей в
   `База_знаний/Инструменты/` (или `Паттерны/`, если это вывод, а не факт про инструмент),
   со статусом "не протестировано" по умолчанию, пока пользователь не подтвердит проверку на практике.
4. Не путать "нашёл во внешнем источнике" и "проверено на наших кампаниях" — это разные статусы.

## Правило бэклога идей

Во время любого обсуждения (по клиенту или по общим концепциям) — если всплывает идея,
гипотеза или направление для будущего мозгового штурма, которое не является прямым
предметом текущего вопроса: не терять её в чате, а сразу дописать в `Бэклог_идей.md`
короткой записью с датой, контекстом (какой клиент/тема породили) и почему это может
быть ценно. Не спрашивать разрешения на каждую такую запись — просто делать по ходу и
кратко упоминать пользователю, что записал. Когда пользователь спрашивает "что у нас есть
интересного на подумать" — читать этот файл целиком и предлагать выжимку.

## Что фиксировать как "решение" vs "лог"

- `Решения.md` — стратегический уровень: почему выбрали направление, что поменяли в
  стратегии и зачем, реакция на результаты по крупным срезам.
- Логи кампаний — операционный уровень: конкретные правки (ставки, объявления, минус-слова)
  и их непосредственный эффект.
- Не дублировать одно и то же в обоих местах — операционное действие попадает в
  `Решения.md` только если оно стратегически значимо (смена подхода, а не рутинная правка).

## Периодический lint (по запросу пользователя)

Проверять: противоречия между `Решения.md` и текущими настройками кампаний; семантику/
аудитории без статуса; orphan-страницы в `База_знаний/`; устаревшие практики, которые
новые источники опровергли; кампании без записей дольше N недель (возможно, заброшены).

## Стиль

Обновлять, а не переписывать заново — вики растёт инкрементально. Ссылки между файлами —
через `[[Имя_страницы]]`. Все новые страницы — с YAML frontmatter минимум `status` и
`updated`, где это применимо.
